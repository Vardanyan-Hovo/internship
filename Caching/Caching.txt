11)Caching is the ability to store copies of frequently accessed data in several places along the request-response path.
    Քեշավորումը հաճախակի հասանելի տվյալների կրկնօրինակները պահելու հնարավորությունն է հարցում-պատասխան ճանապարհի երկայնքով մի քանի վայրերում:

    a)When a consumer requests a resource representation, the request goes through a cache or a series of caches 
        (local cache, proxy cache, or reverse proxy) toward the service hosting the resource.
    b)By using HTTP headers, an origin server indicates whether a response can be cached and, if so, by whom, and for how long.
        Օգտագործելով HTTP վերնագրերը, ծագման սերվերը ցույց է տալիս, թե արդյոք պատասխանը կարող է պահվել քեշում և, եթե այո, ում կողմից և որքան ժամանակով
    c)Caches along the response path can take a copy of a response, but only if the caching metadata allows them to do so.
        Արձագանքման ճանապարհի քեշերը կարող են վերցնել պատասխանի պատճենը, բայց միայն այն դեպքում, եթե քեշավորման մետատվյալները թույլ են տալիս դա անել:

12)Caching in REST APIs
    Being cacheable is one of the architectural constraints of REST.

    a)GET requests should be cachable by default – until a special condition arises. Usually,
        browsers treat all GET requests as cacheable.
        հարցումները լռելյայն պետք է քեշավորվեն՝ մինչև հատուկ պայման առաջանա: Սովորաբար, բրաուզերները 
        վերաբերվում են բոլոր GET հարցումներին, որոնք կարող են քեշավորվել:
    b)POST requests are not cacheable by default but can be made cacheable if either an Expires header or a Cache-Control header with a directive,
         to explicitly allows caching, is added to the response.
        Запросы POST не кэшируются по умолчанию, но их можно сделать кэшируемыми, если к ответу добавлен заголовок "Expires" или заголовок 
        "Cache-Control" с директивой, явно разрешающей кэширование.
    c)Responses to PUT and DELETE requests are not cacheable at all.
    d)HTTP dates are always expressed in GMT, never in local time.
13)Cache Control Headers
    Below given are main HTTP response headers that we can use to "Control Caching Behavior:"

    a)Expires //Ժամկետը լրանում է
        The Expires HTTP header specifies an absolute expiry time for a cached representation. Beyond that time, a cached representation 
        is considered stale and must be re-validated with the origin server. To indicate that a representation never expires, a service can 
        include a time up to one year in the future.
    <--  Expires: Fri, 18 Jul 2023 20:00:00 GMT"  -->

    b) Cache-Control
            The header value comprises(ներառում է) one or more comma-separated directives(հրահանգներ:). These directives determine whether a response is cacheable, and if so, 
            by whom, and for how long e.g. max-age or s-maxage directives.
            Այս հրահանգները որոշում են, թե արդյոք պատասխանը կարող է քեշավորվել, և եթե այո, ում կողմից և որքան ժամանակով, օրինակ. max-age կամ s-maxage հրահանգներ:
    <--  Cache-Control: max-age=3600 -->

    c)Cacheable responses (whether to a GET or to a POST request) should also include a validator — either an ETag or a Last-Modified header.
        Քեշավորվող պատասխանները (լինի GET-ի, թե POST հարցման) պետք է ներառեն նաև վավերացնող՝ կա՛մ ETag, կա՛մ վերջին փոփոխված վերնագիր

14)ETag
    это непрозрачный строковый токен, который сервер связывает с ресурсом, чтобы однозначно идентифицировать состояние ресурса на протяжении его жизненного цикла.

    Если ресурс по данному URL-адресу изменится, необходимо сгенерировать новое значение Etag. Их сравнение может определить, являются ли два представления ресурса одинаковыми.

    При запросе ресурса клиент отправляет серверу ETag в поле заголовка If-None-Match. Сервер соответствует Etag запрошенного ресурса и отправленному значению.
    в заголовке If-None-Match. Если оба значения совпадают, сервер отправляет обратно статус 304 Not Modified без тела, который сообщает клиенту, что кэшированная версия
    ответ все еще можно использовать (свежий).
